# UAT Guide

## 文档目标

这份文档用于指导 Lan 在微信开发者工具中完成一轮 **SQLite 落地后的人工验收**。

本轮验收重点不是看代码，而是确认以下两件事：
1. **主业务闭环可用**：顾客申请 -> 店员审核 -> 顾客查看结果
2. **数据可持久化**：服务重启后规则、预约、审核结果不会丢失

---

## 一、验收前准备

### 1. 启动后端

在项目根目录执行：

```bash
npm run dev:server
```

可选：如果你想显式指定店员白名单与数据库文件路径，可以这样启动：

```bash
STAFF_OPEN_IDS=staff-openid-demo SQLITE_PATH=/absolute/path/miniapp.sqlite npm run dev:server
```

默认情况下：
- 服务地址：`http://127.0.0.1:3000`
- 数据库文件：`apps/server/data/miniapp.sqlite`
- 店员白名单默认值：`staff-openid-demo`

### 2. 可选：先跑后端自测

```bash
npm run test:server
```

如果输出：

```text
self-test ok
```

说明后端的 SQLite 持久化与核心接口链路已通过机器自测。

### 3. 打开微信开发者工具

- 打开目录：`apps/weapp`
- 勾选开发环境中允许本地接口联调的相关选项
- 确认小程序请求的 API 基地址仍是：`http://127.0.0.1:3000`

### 4. 准备店员测试身份

店员页会要求输入一个 OpenID。

本地联调时，可直接输入：

```text
staff-openid-demo
```

如果你启动后端时自定义了 `STAFF_OPEN_IDS`，则要输入你自己配置进去的值。

## 二、建议验收顺序

建议按下面顺序操作，不要跳步：

1. 首页返图展示
2. 顾客提交预约申请
3. 顾客查看“我的预约”
4. 店员修改预约规则
5. 店员审核预约
6. 顾客再次查看状态变化
7. 重启后端验证 SQLite 持久化

---

## 三、详细验收步骤

## Case 1：首页返图展示

### 操作
1. 打开首页 `pages/home`
2. 观察页面是否展示：
   - 品牌欢迎区
   - 返图区
   - “立即预约”入口
   - “查看我的预约”入口

### 预期结果
- 页面不白屏
- 至少看到 3 条返图数据
- 返图顺序稳定
- 首页风格不是旧版“服务项目列表”样式

### 若失败，记录什么
- 是空白、报错、还是仅文案不对
- 是否有 toast 提示
- 截图首页状态

---

## Case 2：顾客提交合法预约申请

### 操作
1. 进入预约页 `pages/booking`
2. 选择一个可预约日期
3. 选择一个可预约时间段
4. 输入：
   - 姓名：`Lan`
   - 手机号：`13900000001`
   - 备注：`UAT-顾客首次预约`
5. 提交预约

### 预期结果
- 提交成功
- 页面提示“已提交，待审核”或等价文案
- 提交后跳转到“我的预约”页，或可手动进入“我的预约”页查看
- 新记录状态为：`待审核`

### 若失败，记录什么
- 失败时的 toast 文案
- 选择的日期 / 时间段
- 输入的手机号
- 截图

---

## Case 3：顾客查看“我的预约”

### 操作
1. 进入 `pages/my-bookings`
2. 若页面未自动带出手机号，则手动输入：`13900000001`
3. 点击查询

### 预期结果
- 能查到刚刚创建的预约
- 状态文案是中文：`待审核`
- 不应直接展示英文原始值 `pending`
- 若有备注或审核说明，可正常展示

### 若失败，记录什么
- 是查不到，还是状态错了
- 是否出现接口错误提示
- 截图

---

## Case 4：店员修改预约规则

### 操作
1. 进入 `pages/staff/rules`
2. 输入店员 OpenID：`staff-openid-demo`
3. 加载成功后，修改规则为：
   - `advanceOpenDays = 1`
   - 新增一个关闭日期，例如：`2026-03-20`
   - 设置时间段：
     - `09:30-10:30`
     - `11:00-12:00`
4. 点击保存

### 预期结果
- 页面提示“规则已保存”
- 重新进入或下拉刷新后，规则仍保持刚才保存的值
- 不会被自动还原成旧默认值

### 若失败，记录什么
- 保存失败还是刷新后丢失
- 是否提示无权限
- 截图保存前后状态

---

## Case 5：店员审核预约

### 操作
1. 进入 `pages/staff/appointments`
2. 输入店员 OpenID：`staff-openid-demo`
3. 默认查看 `待审核` 列表
4. 找到手机号 `13900000001` 对应的预约
5. 在审核备注中填写：`请准时到店`
6. 点击“通过”

### 预期结果
- 页面提示“已通过”
- 列表刷新后，该条预约不再出现在 `待审核` 中
- 如果切换到 `已通过`，可以看到该记录

### 若失败，记录什么
- 是否报 `该时间段已被占用`
- 是否报 `该预约已审核`
- 是否报无权限
- 截图

---

## Case 6：顾客查看审核后状态

### 操作
1. 回到 `pages/my-bookings`
2. 输入手机号：`13900000001`
3. 点击查询

### 预期结果
- 该条预约状态从 `待审核` 变成 `已通过`
- 审核备注如已展示，则可看到：`请准时到店`

### 若失败，记录什么
- 是否仍停留在待审核
- 是否查不到记录
- 截图

---

## Case 7：重启后端后验证 SQLite 持久化

### 操作
1. 保留刚才已经创建并审核通过的数据
2. 停掉后端服务
3. 重新执行：

```bash
npm run dev:server
```

4. 重启后再次执行以下检查：
   - 首页返图仍能正常加载
   - `pages/staff/rules` 中仍是刚才保存的新规则
   - `pages/my-bookings` 中手机号 `13900000001` 仍能查到已通过记录

### 预期结果
- 规则不丢失
- 预约记录不丢失
- 审核结果不丢失
- availability 仍受新规则影响

### 若失败，记录什么
- 是规则丢了、预约丢了，还是状态回退了
- 重启前后截图

## 四、建议补充验证（可选但推荐）

## Case 8：无权限店员访问

### 操作
1. 进入 `pages/staff/rules` 或 `pages/staff/appointments`
2. 输入一个不在白名单里的值，例如：`not-allowed-openid`

### 预期结果
- 页面提示：`当前账号无店员权限`
- 不应返回规则数据或预约列表

---

## 五、验收通过标准

本轮可以认为“通过”的最低标准：

- 顾客能成功提交预约申请
- 顾客能查到自己的预约状态
- 店员能成功修改预约规则
- 店员能成功审核预约
- 审核后的状态能回写到顾客侧
- 重启服务后，规则 / 预约 / 审核结果仍然保留
